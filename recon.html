WHY THIS WORKS
// WHY DNS REVEALS M365

DNS records are public — they are required for email routing and authentication to function, so every mail system must publish them. This makes them a reliable passive fingerprinting surface with no interaction required.

MX RECORDS
Reveal where a domain receives mail. Microsoft uses a predictable hostname for Exchange Online: [tenant].mail.protection.outlook.com

SPF RECORD
Lists authorized sending servers. M365 tenants always include spf.protection.outlook.com in their SPF TXT record.

DKIM CNAMEs
The strongest M365 signal. When DKIM is enabled in the M365 admin, two CNAME records are created that always resolve to *.onmicrosoft.com — a Microsoft-exclusive namespace. These only exist if DKIM was explicitly configured by an admin.

DMARC POLICY
Queries _dmarc.domain.com. A p=reject policy requires significant admin effort and strongly correlates with active security management and Defender deployment.

MTA-STS
Queries _mta-sts.domain.com. Mail Transport Agent Strict Transport Security is an advanced config rarely seen without dedicated security tooling. Almost never present on basic M365 tenants.

// WHAT DNS CANNOT PROVE

DNS confirms Exchange Online is in use. It cannot confirm which Microsoft 365 license tier the org bought, or whether Defender for Office 365 is specifically enabled. Defender is a licensing decision made inside Microsoft's admin portal — it leaves no unique DNS record.

Third-party gateways: If the org routes mail through Mimecast, Proofpoint, or another SEG before M365, the MX record points to that vendor — causing a false negative on check [1] even though M365 is used internally.

WHY HEADERS REVEAL DEFENDER:
When Defender for Office 365 processes a message, Microsoft injects specific headers ONLY present with an active Defender license:

X-MS-Exchange-AtpMessageProperties
Contains SA (Safe Attachments) and/or SL (Safe Links). This header is NEVER set by plain Exchange Online / EOP. It is the most definitive external Defender signal.

UIMP in X-Forefront-Antispam-Report
User Impersonation Protection field — exclusive to Defender for Office 365 anti-phishing policies.

Safe Links URL wrapping
URLs rewritten to *.safelinks.protection.outlook.com. Only occurs when a Defender Safe Links policy is active on the recipient tenant.

// Headers are stamped by the RECIPIENT's system — you need email FROM the target tenant.
